# How It Works — Plain English

> 📖 **This file is for humans.** It's a non-technical explainer of what the Shipwreck Blog Engine does, why it's lighter than WordPress, and what trade-offs you accept. Read it once before you integrate, share it with anyone curious about why we don't use WordPress.
>
> Agents: this isn't a runbook. If you're integrating, go to [AGENTS.md](AGENTS.md). If you're explaining the project to the user, you can quote from this file freely.
>
> _Last reviewed for engine version: 0.3.5_

## Three pieces, none of them on the hosting server

1. **The engine** — a set of npm packages (`@shipwreck/blog-core`, `@shipwreck/blog-theme-default`) plus the agent integration skill. Lives in this GitHub repo. Defines schemas, components, page templates, SEO helpers.
2. **The site source** — your site's git repo with `_blog/` inside it. Contains your `tokens.css`, your header/footer, your MDX posts, your author files. This is where editing happens.
3. **The built blog** — a folder of plain HTML/CSS/image files generated by running `npm run build`. This is what ends up on the hosting server.

**Building** happens once on your local machine or in CI: Node reads your MDX posts + the engine's templates + your theme tokens, and writes out static HTML. Each post becomes one `index.html`. Each tag/category/author archive becomes one `index.html`. Sitemap, RSS, robots — all generated as plain XML/text files. Total output: a folder of static files, typically under 5 MB for a small site.

**Deploying** is just copying that folder to wherever the host serves `/blog/` from. Same as uploading any static page.

**At runtime**, the hosting server's entire job is to read a file off disk and send the bytes. No PHP, no database, no Node, no Astro, no Tailwind — none of those run on the host. They all ran on the build machine, the output is plain text/CSS/images.

## Zero database

**No MySQL/MariaDB/Postgres. No SQLite. No Redis. No anything.** The host stores files; that's it.

What this means concretely vs. a WordPress blog on the same server:

| Per-request work | WordPress | Shipwreck Blog |
|---|---|---|
| PHP interpreter boot | yes (per request unless OPcache) | none |
| Database connection + query | yes (often 30+ queries per page) | none |
| Theme PHP rendering | yes | none |
| Plugin chain execution | yes (often 20+ plugins) | none |
| Disk I/O | many file includes + plugin reads | one `index.html` read |
| Typical TTFB (cold) | 200–800ms | 5–20ms |
| Typical TTFB (warm cache) | 50–200ms | <5ms (often kernel page cache hit) |

A WordPress page request walks through a stack worth ~50,000 lines of PHP across core + plugins. A Shipwreck Blog page request walks through `read('/home/.../blog/<slug>/index.html')` — kernel-level. The difference at scale is enormous: a $5 cPanel account that struggles with 100 concurrent WordPress visitors will trivially serve thousands of concurrent static-blog visitors because there's literally no work being done per request.

## What you give up (be honest)

- **Edits aren't live until rebuild.** Add a post → rebuild → redeploy. Sveltia CMS at `/blog/admin/` makes the editing UX feel like WordPress (browser form, hit save), but "save" triggers a git commit which triggers a rebuild via your host's deploy pipeline. Latency is whatever the deploy takes — typically seconds to a few minutes vs. WordPress's "instant publish."
- **No live comments.** If you want them, bolt on Disqus / Giscus / whatever via embed. Not core to the engine.
- **No dynamic personalization.** No "logged-in user sees X." The site is the same for everyone (which is what 99% of blogs actually need anyway).

For a content blog (which is the use case), this is a massive win. For a transactional app pretending to be a blog, you'd want a different tool.

---

## The compounding benefit

A static `/blog/` next to a WordPress apex is the best of both. WordPress keeps doing what only WordPress can do (commerce, dynamic content, plugins). The blog stops competing for PHP workers, memory, and database connections — those are now reserved for the parts of the site that genuinely need them. Sites that bog down at 50 concurrent users on shared cPanel because WordPress is grinding through the blog page typically gain back 30–50% of capacity by static-ifying the blog alone.

The other compounding benefit: the static `/blog/` is **massively easier to cache aggressively at the CDN edge.** A Cloudflare cache rule with `cache_everything` + `edge_ttl=86400` on `/blog/*` means most users never touch your origin at all. WordPress can never safely do this without breaking logged-in admin paths, search, comments, etc. Static can.

## What this looks like for someone editing the blog

If you're a non-developer adding a post:

1. Open `https://<your-site>/blog/admin/` in a browser
2. Log in with your GitHub account (one-time setup per site)
3. Click "New Post", fill in the form (title, body, category, tags, FAQ items, etc.) — looks like a WordPress post editor
4. Click Save → a git commit happens behind the scenes
5. Your host's deploy pipeline picks up the commit, rebuilds the blog, publishes it
6. Refresh `/blog/<your-slug>/` — your post is live

The editor is [Sveltia CMS](https://github.com/sveltia/sveltia-cms) — open-source, free, no database, no separate server. Your posts are plain MDX files in your site repo. If Sveltia ever disappears, your content is still git-tracked text files you can edit by hand.

If you're a developer or AI agent adding a post: write the MDX directly under `_blog/src/content/posts/<slug>.mdx`, commit, push. Same end result. The agent runbook for this is [.claude/skills/add-shipwreck-blog-post.md](.claude/skills/add-shipwreck-blog-post.md).

## Where the build runs (and why this matters)

You can build the blog in any of these places — pick whichever fits the host:

- **On your laptop** — `cd _blog && npm run build && cp -r dist ../blog`. Then commit + push, or rsync the `blog/` dir to the server. Works for any host.
- **In CI on a host that auto-deploys from git** — Cloudflare Pages, Netlify, Vercel, GitHub Pages all auto-build the blog when they see `_blog/` in your commits. Zero work after the initial integration.
- **On the host server itself** — only useful if the host has Node 20+. Most cheap shared hosts don't, but VPS / dedicated hosts can if you want.

The "where" doesn't matter to the visitor. The blog is the same static files in all three cases. Pick whatever's easiest for the host's existing workflow.

## Why we made it this way

The original brief was: "give every site a blog without WordPress, without lock-in, without per-site infrastructure to maintain." Static-first delivers all three:

- **Without WordPress** — no PHP, no database, no plugins, no security update treadmill, no theme bloat
- **Without lock-in** — your content is plain MDX text files in your own git repo. Engine could disappear tomorrow and your posts would still be readable
- **Without per-site infrastructure** — the host site already deploys somehow. The blog rides along on that. There's no "blog server" to maintain

The trade-off (rebuild-to-publish) is genuinely a trade-off — but for blog content, where most posts go up once and rarely change, it's the right one. WordPress optimizes for "change content live"; we optimize for "serve content fast and cheap." Pick the right tool for the actual usage pattern.

---

## When NOT to use this

- The site needs comments with live moderation, real-time updates, or user accounts that interact with content (forums, members-only areas)
- The site is genuinely an app where the blog is incidental (e.g., a SaaS dashboard with a tiny blog tab — easier to use the SaaS's existing tooling)
- The team editing content has zero git tolerance and won't use Sveltia CMS — though Sveltia is designed to look exactly like WordPress, so this is rarer than people assume

For everything else (marketing blogs, news sites, doc sites, personal sites, agency portfolios with a blog), static-first wins on speed, cost, security surface, and operator burden.

## Glossary (for non-developers)

- **Static HTML** — plain text files the server reads and sends. The opposite of "dynamic" pages built per request by a database+code combination.
- **MDX** — Markdown plus inline components. Looks like Markdown for normal writing; lets you embed richer elements when needed.
- **TTFB** — "Time To First Byte." How long the server takes to start sending data after the visitor's request. Lower is faster.
- **CDN** — Content Delivery Network (Cloudflare, Fastly, etc.). A globally-distributed cache that serves your files from the visitor's nearest geographic location.
- **OPcache** — PHP's bytecode cache. Without it, every WordPress request re-parses thousands of PHP files.
- **Sitemap** — a machine-readable list of every URL on your site, for search engines.
- **`/blog/`** — the URL path the blog mounts at on the host site. e.g. `wollongongweather.com/blog/`. Locked in once chosen — don't change it after launch.
